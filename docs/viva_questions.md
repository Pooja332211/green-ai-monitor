# Viva Questions: Green AI Monitor Project

## Project Overview Questions

1. **What is the primary objective of the Green AI Monitor project?**
   - Expected Answer: To track and monitor carbon emissions generated by AI/ML workloads in real-time, providing transparency and tools for sustainable AI practices.

2. **Why is monitoring carbon emissions in AI workloads important?**
   - Expected Answer: AI systems consume significant energy, contributing to global carbon emissions. Monitoring helps developers understand and minimize their environmental impact.

3. **What are the main components of the Green AI Monitor system?**
   - Expected Answer: Backend API (FastAPI), Frontend Dashboard (React), Carbon Tracking Service, System Monitoring Service, Database (SQLite), Chatbot, and Sample ML Workloads.

## Technical Architecture Questions

4. **Explain the layered architecture used in this project.**
   - Expected Answer: The system uses a layered architecture with Presentation (Frontend), Application (Backend API), Domain (Services), and Infrastructure (Database) layers for clear separation of concerns.

5. **How does the carbon tracking service work?**
   - Expected Answer: It uses the CodeCarbon library to track emissions based on energy consumption. It starts/stops tracking sessions for different projects and calculates CO2 emissions in real-time.

6. **What technology stack is used for the backend?**
   - Expected Answer: Python with FastAPI for the web framework, SQLAlchemy for ORM, and SQLite for the database.

7. **How is the frontend implemented and what are its key features?**
   - Expected Answer: Built with React.js, featuring a dashboard with live stats, emission charts, and optimization tips. It communicates with the backend via REST APIs.

## Implementation Details Questions

8. **How are emissions calculated in the system?**
   - Expected Answer: Using the CodeCarbon library, which estimates CO2 emissions based on hardware power consumption, location-based carbon intensity, and usage duration.

9. **Explain the role of the system monitoring service.**
   - Expected Answer: It collects hardware metrics like CPU usage, memory consumption, and power draw to correlate system performance with carbon emissions.

10. **How does the chatbot component work?**
    - Expected Answer: Uses NLP techniques with NLTK for text processing, matches user input against predefined intents in intents.json, and applies rule-based reasoning for contextual responses.

11. **What is the purpose of the sample ML workloads in the project?**
    - Expected Answer: To demonstrate emission tracking on real AI tasks like BERT fine-tuning, LSTM training, ResNet training, and YOLO inference.

## Database and API Questions

12. **Describe the database schema used in the project.**
    - Expected Answer: Uses SQLAlchemy models for CarbonEmission, SystemMetrics, and Log tables. CarbonEmission stores emission data, SystemMetrics tracks hardware usage, and Log records system events.

13. **What are the main API endpoints and their functions?**
    - Expected Answer: /api/v1/tracking (start/stop emission tracking), /api/v1/stats (retrieve statistics), /api/v1/logs (access logs), /api/v1/suggestions (get optimization tips).

14. **How is data validation handled in the API?**
    - Expected Answer: Using Pydantic schemas for request/response validation, ensuring data integrity and providing automatic documentation.

## Challenges and Solutions Questions

15. **What challenges did you face in integrating real-time emission tracking?**
    - Expected Answer: Balancing accuracy with system performance impact, handling different hardware configurations, and ensuring reliable data collection during ML workloads.

16. **How does the system handle concurrent tracking sessions?**
    - Expected Answer: The CarbonTrackerService prevents multiple concurrent sessions by checking if a tracker is already active before starting a new one.

17. **What measures are taken for system performance optimization?**
    - Expected Answer: Asynchronous operations for monitoring, efficient database queries, caching of frequently accessed data, and batch operations for data storage.

## Future Enhancements Questions

18. **What are potential future enhancements for this project?**
    - Expected Answer: Multi-user support, cloud provider integrations, predictive emission analysis using ML, integration with CI/CD pipelines, and support for additional emission tracking libraries.

19. **How could this system be scaled for enterprise use?**
    - Expected Answer: Migrate to production databases (PostgreSQL), implement authentication/authorization, add containerization (Docker/Kubernetes), and develop admin dashboards for multi-user management.

20. **What role could machine learning play in improving the system?**
    - Expected Answer: Predictive modeling for emission forecasting, anomaly detection in emission patterns, automated optimization recommendations, and intelligent chatbot responses.

## Environmental Impact Questions

21. **How does this project contribute to sustainable AI development?**
    - Expected Answer: By providing transparency into AI's carbon footprint, enabling data-driven decisions for emission reduction, and educating users about environmental impacts.

22. **What are some practical ways AI workloads can reduce their carbon emissions?**
    - Expected Answer: Use energy-efficient hardware, optimize model architectures, leverage cloud providers with renewable energy, implement model compression, and schedule training during off-peak hours.

23. **How accurate is the emission tracking in this system?**
    - Expected Answer: Accuracy depends on CodeCarbon's methodology, which considers hardware specifications, energy sources, and regional carbon intensity. It's an estimation tool providing valuable insights rather than exact measurements.
